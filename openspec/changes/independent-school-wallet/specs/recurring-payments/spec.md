# Spec: Recurring Payments & Scheduling Engine

## Overview
The `ScheduledPayment` model represents a payment intent — a promise to execute a payment at a future date. A background cron job processes due payments, handles failures with exponential backoff retries, and auto-generates next occurrences for recurring fees. This spec covers the model, API, scheduling engine, and all edge cases.

---

## Model: `ScheduledPayment`

### Schema

| Field | Type | Required | Default | Constraints |
|-------|------|----------|---------|-------------|
| `parent` | ObjectId → User | yes | — | set from `req.userId`, immutable |
| `child` | ObjectId → Child | yes | — | must belong to parent |
| `feeCategory` | ObjectId → FeeCategory | yes | — | must belong to parent |
| `amount` | Number | yes | — | snapshot from fee category, min: 50 |
| `scheduledDate` | Date | yes | — | must be future on creation |
| `status` | Enum | no | `"pending"` | `pending`, `processing`, `completed`, `failed`, `cancelled`, `skipped` |
| `retryCount` | Number | no | `0` | max: 3 |
| `lastRetryAt` | Date | no | — | timestamp of last retry attempt |
| `failureReason` | String | no | — | human-readable failure description |
| `transaction` | ObjectId → Transaction | no | — | populated after successful execution |
| `idempotencyKey` | String | auto | — | `${_id}-${scheduledDate.toISOString()}` |
| `isAutoGenerated` | Boolean | no | `false` | true for occurrences created by the engine |
| `sourcePayment` | ObjectId → ScheduledPayment | no | — | link to the original payment that spawned this occurrence |
| `version` | Number | no | `0` | optimistic concurrency control |
| `createdAt` | Date | auto | `Date.now` | — |
| `updatedAt` | Date | auto | `Date.now` | — |

### Status State Machine

```
                    ┌──────────┐
                    │  pending  │ ← initial state / re-queued after retry
                    └─────┬────┘
                          │
                    ┌─────▼────┐
               ┌────│processing │────┐
               │    └──────────┘    │
               │                    │
         ┌─────▼─────┐      ┌──────▼────┐
         │ completed  │      │  failed   │
         └───────────┘      └─────┬─────┘
                                  │
                          retryCount < 3?
                          ├── yes → pending (re-queued with backoff)
                          └── no  → failed (terminal)

  External actions:
    pending → cancelled  (user cancels)
    pending → skipped    (user skips occurrence)
    pending → processing (user triggers pay-now)
```

### Indexes
- `{ status: 1, scheduledDate: 1 }` — cron job: find due payments
- `{ parent: 1, status: 1 }` — parent's pending/active payments
- `{ child: 1, feeCategory: 1, scheduledDate: 1 }` — unique compound to prevent duplicate scheduling
- `{ feeCategory: 1, status: 1 }` — find payments for a fee category (used when deactivating fees)
- `{ idempotencyKey: 1 }` — unique, prevents double-execution

### Mongoose Options
- `timestamps: true`
- `optimisticConcurrency: true` (uses `version` field)

## API Endpoints

### `POST /api/scheduled-payments`
**Purpose:** Create a scheduled payment for a child + fee category.

**Auth:** Required (parent role).

**Request body:**
```json
{
  "child": "<childId>",
  "feeCategory": "<feeCategoryId>",
  "scheduledDate": "2026-04-15T00:00:00.000Z"
}
```

**Validation rules:**
- `child`: required, valid ObjectId, must belong to `req.userId`
- `feeCategory`: required, valid ObjectId, must belong to `req.userId`
- `scheduledDate`: required, ISO 8601, must be in the future
- Child must belong to the same school profile as the fee category
- Amount is auto-set from `feeCategory.amount` (snapshot at creation time)
- No existing pending/processing scheduled payment for same `{ child, feeCategory, scheduledDate }`
- Business rule: max 100 pending scheduled payments per parent

**Happy path (201):**
```json
{
  "message": "Payment scheduled",
  "scheduledPayment": {
    "_id": "...",
    "child": { "firstName": "Chinedu", "lastName": "Okafor" },
    "feeCategory": { "name": "Term 2 Tuition", "amount": 75000 },
    "amount": 75000,
    "scheduledDate": "2026-04-15T00:00:00.000Z",
    "status": "pending"
  }
}
```

**Unhappy paths:**
| Status | Condition | Response |
|--------|-----------|----------|
| 400 | Missing required fields | `{ "errors": [...] }` |
| 400 | Child not at same school as fee category | `{ "error": "Child is not enrolled at the school for this fee category" }` |
| 400 | `scheduledDate` in the past | `{ "error": "Scheduled date must be in the future" }` |
| 409 | Duplicate scheduled payment | `{ "error": "A payment is already scheduled for this child and fee on this date" }` |
| 429 | 100 pending payments | `{ "error": "Maximum pending scheduled payments reached (100)" }` |

### `POST /api/scheduled-payments/bulk`
**Purpose:** Create scheduled payments for all applicable children of a fee category.

**Request body:**
```json
{
  "feeCategory": "<feeCategoryId>",
  "scheduledDate": "2026-04-15T00:00:00.000Z"
}
```

**Logic:**
1. Get fee category and its `applicableTo` children (or all active children at school if empty)
2. For each child, create a `ScheduledPayment` (skip if duplicate exists)
3. Return created count and skipped count

**Response (201):**
```json
{
  "message": "Payments scheduled",
  "created": 2,
  "skipped": 1,
  "scheduledPayments": [...]
}
```

### `PUT /api/scheduled-payments/:id/skip`
**Purpose:** Skip this occurrence. If recurring, generates the next one.

**Pre-conditions:** Status must be `pending`.

**Side effects:**
1. Set status to `skipped`
2. If the source fee category `isRecurring`, compute and create next occurrence

**Response (200):**
```json
{
  "message": "Payment skipped",
  "scheduledPayment": { ..., "status": "skipped" },
  "nextOccurrence": {
    "_id": "...",
    "scheduledDate": "2026-08-15T00:00:00.000Z",
    "status": "pending"
  }
}
```

---

### `POST /api/scheduled-payments/:id/pay-now`
**Purpose:** Immediately execute a scheduled payment (bypasses the cron wait).

**Pre-conditions:** Status must be `pending`.

**Logic:**
1. Acquire optimistic lock (set status to `processing`, increment version)
2. Check parent balance
3. If sufficient: create Transaction, deduct balance, set status to `completed`
4. If insufficient: set status to `failed`, set failure reason
5. If recurring and completed: generate next occurrence

**Response (200):**
```json
{
  "message": "Payment executed successfully",
  "scheduledPayment": { ..., "status": "completed" },
  "transaction": { "_id": "...", "amount": 75000, "reference": "TXN-..." },
  "newBalance": 425000,
  "nextOccurrence": { ... }
}
```

**Unhappy paths:**
| Status | Condition | Response |
|--------|-----------|----------|
| 400 | Status is not `pending` | `{ "error": "Can only execute pending payments" }` |
| 400 | Insufficient balance | `{ "error": "Insufficient balance. Required: ₦75,000. Available: ₦50,000" }` |
| 409 | Optimistic lock conflict | `{ "error": "Payment is being processed. Please try again." }` |

## Scheduling Engine (Background Cron)

### Architecture

```
┌─────────────────────────────────────────────────┐
│                 Cron Job (every 15 min)          │
│                                                   │
│  1. Query: { status: 'pending',                  │
│              scheduledDate: { $lte: now } }       │
│     Limit: 50 per batch                          │
│                                                   │
│  2. For each payment:                            │
│     a. Acquire lock (optimistic concurrency)     │
│     b. Check balance                             │
│     c. Execute or fail                           │
│     d. Generate next occurrence if recurring     │
│                                                   │
│  3. Log results                                  │
│     { processed: N, succeeded: M, failed: K }    │
└─────────────────────────────────────────────────┘
```

### Implementation: `services/paymentScheduler.js`

```javascript
// Pseudocode for the scheduling engine

const BATCH_SIZE = 50;
const MAX_RETRIES = 3;
const RETRY_DELAYS = [60 * 60 * 1000, 6 * 60 * 60 * 1000, 24 * 60 * 60 * 1000]; // 1h, 6h, 24h

async function processDuePayments() {
  const now = new Date();
  
  const duePayments = await ScheduledPayment.find({
    status: 'pending',
    scheduledDate: { $lte: now }
  })
  .sort({ scheduledDate: 1 })
  .limit(BATCH_SIZE)
  .populate('parent child feeCategory');

  const results = { processed: 0, succeeded: 0, failed: 0 };

  for (const payment of duePayments) {
    try {
      // 1. Acquire lock
      const locked = await ScheduledPayment.findOneAndUpdate(
        { _id: payment._id, status: 'pending', version: payment.version },
        { $set: { status: 'processing' }, $inc: { version: 1 } },
        { new: true }
      );
      if (!locked) continue; // Already processing

      // 2. Execute payment
      const result = await executePayment(locked);
      results.processed++;

      if (result.success) {
        results.succeeded++;
        // Generate next occurrence if recurring
        await generateNextOccurrence(locked);
      } else {
        results.failed++;
        await handleFailure(locked, result.reason);
      }
    } catch (error) {
      console.error(`Error processing payment ${payment._id}:`, error);
      await ScheduledPayment.findByIdAndUpdate(payment._id, {
        $set: { status: 'pending' } // Reset to pending on unexpected error
      });
    }
  }

  return results;
}

async function executePayment(scheduledPayment) {
  const parent = await User.findById(scheduledPayment.parent);
  
  if (parent.balance < scheduledPayment.amount) {
    return { success: false, reason: 'Insufficient balance' };
  }

  // Atomic balance deduction
  const updated = await User.findOneAndUpdate(
    { _id: parent._id, balance: { $gte: scheduledPayment.amount } },
    { $inc: { balance: -scheduledPayment.amount } },
    { new: true }
  );

  if (!updated) {
    return { success: false, reason: 'Insufficient balance (race condition)' };
  }

  // Create transaction
  const transaction = new Transaction({
    user: parent._id,
    type: 'payment',
    amount: scheduledPayment.amount,
    status: 'completed',
    reference: `SCH-${Date.now()}-${Math.random().toString(36).substr(2, 9)}`,
    paymentMethod: 'auto',
    metadata: {
      scheduledPaymentId: scheduledPayment._id,
      childId: scheduledPayment.child,
      feeCategoryId: scheduledPayment.feeCategory
    }
  });
  await transaction.save();

  // Mark completed
  await ScheduledPayment.findByIdAndUpdate(scheduledPayment._id, {
    $set: { status: 'completed', transaction: transaction._id }
  });

  return { success: true, transaction };
}

async function handleFailure(payment, reason) {
  const retryCount = payment.retryCount + 1;
  
  if (retryCount >= MAX_RETRIES) {
    // Terminal failure
    await ScheduledPayment.findByIdAndUpdate(payment._id, {
      $set: {
        status: 'failed',
        failureReason: reason,
        retryCount,
        lastRetryAt: new Date()
      }
    });
  } else {
    // Re-queue with backoff
    const delay = RETRY_DELAYS[retryCount - 1];
    await ScheduledPayment.findByIdAndUpdate(payment._id, {
      $set: {
        status: 'pending',
        failureReason: reason,
        scheduledDate: new Date(Date.now() + delay),
        retryCount,
        lastRetryAt: new Date()
      }
    });
  }
}

async function generateNextOccurrence(completedPayment) {
  const feeCategory = await FeeCategory.findById(completedPayment.feeCategory);
  
  if (!feeCategory || !feeCategory.isRecurring || !feeCategory.isActive) return;

  const nextDate = computeNextDate(
    completedPayment.scheduledDate,
    feeCategory.recurrenceRule
  );

  // Check if past end date
  if (feeCategory.recurrenceRule.endDate && nextDate > feeCategory.recurrenceRule.endDate) {
    return; // Recurrence ended
  }

  // Create next occurrence (idempotent via unique index)
  try {
    await ScheduledPayment.create({
      parent: completedPayment.parent,
      child: completedPayment.child,
      feeCategory: completedPayment.feeCategory,
      amount: feeCategory.amount, // Use current fee amount
      scheduledDate: nextDate,
      status: 'pending',
      isAutoGenerated: true,
      sourcePayment: completedPayment._id
    });
  } catch (error) {
    if (error.code !== 11000) throw error; // Ignore duplicate
  }
}
```

### Recurrence Date Calculation

```javascript
function computeNextDate(currentDate, rule) {
  const date = new Date(currentDate);
  
  switch (rule.frequency) {
    case 'weekly':
      date.setDate(date.getDate() + 7);
      break;
    case 'biweekly':
      date.setDate(date.getDate() + 14);
      break;
    case 'monthly':
      date.setMonth(date.getMonth() + 1);
      // Clamp to last day of month if needed
      if (rule.dayOfMonth) {
        const lastDay = new Date(date.getFullYear(), date.getMonth() + 1, 0).getDate();
        date.setDate(Math.min(rule.dayOfMonth, lastDay));
      }
      break;
    case 'termly':
      date.setMonth(date.getMonth() + 4); // ~Nigerian term length
      break;
    case 'annually':
      date.setFullYear(date.getFullYear() + 1);
      break;
  }
  
  return date;
}
```

## Concurrency & Idempotency

### Problem: Double-execution
If two cron instances (or cron + pay-now) process the same payment simultaneously, the balance could be deducted twice.

### Solution: Three-layer protection
1. **Optimistic locking**: `findOneAndUpdate` with `version` field — only one acquirer succeeds
2. **Atomic balance deduction**: `$inc` with `$gte` guard — prevents over-deduction
3. **Idempotency key**: Unique `idempotencyKey` on Transaction creation — prevents duplicate transactions

### Problem: Stale `processing` status
If the server crashes after setting status to `processing` but before completion, the payment is stuck.

### Solution: Stale lock recovery
A secondary cron (every hour) finds payments stuck in `processing` for > 30 minutes and resets them to `pending`:
```javascript
await ScheduledPayment.updateMany(
  {
    status: 'processing',
    updatedAt: { $lt: new Date(Date.now() - 30 * 60 * 1000) }
  },
  { $set: { status: 'pending' } }
);
```

## Notification Hooks (Future)

The scheduling engine emits events at key points for future notification integration:
- `payment.scheduled` — new payment created
- `payment.executed` — payment succeeded
- `payment.failed` — payment failed (each retry)
- `payment.failed.terminal` — payment exhausted retries
- `payment.upcoming` — payment due in 24 hours

These are currently logged to console. Future: emit via EventEmitter or message queue.
